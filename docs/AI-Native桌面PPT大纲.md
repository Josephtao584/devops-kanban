## 第1页：历史重演：纺织业走过的路

- **手工期**：100个织女工亲手织，人均日产1匹布
- **机械化期**：10个工人管机器，人均日产50匹布
- **智能化期**：1个人管系统，人均日产2000匹布

每一次技术跃迁，组织都跟着重构 —— 从人人干活到少数人管多数机器，再到一个人管整个系统。纺织业没有萎缩，反而变成了更大的产业。

**配图描述（尺寸：1920×600px，横向三段式）：**

> 从左到右三个场景渐变过渡——最左：一间昏暗的手工作坊，几十个织女工坐在木织机前密密麻麻，人人低头忙碌。中间：一间机械化车间，几台大型动力织布机整齐排列，少量工人穿行巡视。最右：一间明亮的智能工厂，只有一个人站在中控大屏前，身后是全自动化产线高速运转。色调从左到右由暖黄变为冷蓝，传达"从人力到智能"的时代变迁。

## 第2页：AI驱动的产业变革：与织女工的命运重演

**类比：工业革命 vs AI革命**

| | 工业革命 | AI革命 |
|---|---|---|
| **生产单元** | 织女工 → 动力织布机 | 程序员 → AI Agent |
| **组织形态** | 手工作坊 → 工厂 | 职能型开发团队 → 人机混合团队 |
| **管理对象** | 管人 → 管机器和流程 | 管工程师 → 编排Agent工作流 |
| **新岗位诞生** | 机器操作工、质检员、车间主任 | Agent监督者（Agent Supervisor）、AI编排者（AI Orchestrator）、AI运营（AI Operations） |

**组织变革：As-Is vs To-Be**

| | As-Is（当前） | To-Be（AI-Native） |
|---|---|---|
| **个人** | 程序员写代码，核心能力是编码速度 | AI编排者管Agent，核心能力是判断力和决策力 |
| **团队** | 5人团队全员写代码，串行交付 | 1人 + 多个Agent并行，5人团队+10个Agent产出超20人团队 |
| **管理** | 盯人、盯工时、开周会问进度 | 盯流、盯任务状态、看实时仪表盘 |
| **流程** | 手动排期、手动分配、手动交接 | 自动编排、自动流转、关键节点暂停等人确认 |

**核心论点：** 工业革命没有消灭纺织业，而是催生了现代制造业体系。AI革命不会消灭软件行业，而是催生**AI-Native的组织形态**。两个时代、同一个命题：**被替代还是进化**。

**配图描述（尺寸：960×540px，左右各一张）：**

左图（As-Is — 当前组织）：
> 一个典型的软件团队办公场景：一排程序员坐在工位上，每个人面前是编辑器+终端，都在埋头写代码。左边一个PM拿着文档排队等开发，右边一个QA对着测试报告皱眉。整体拥挤忙碌，墙上贴着Sprint看板，上面全是红色的"进行中"卡片。色调偏灰暗，传达繁忙但低效的感觉。

右图（To-Be — AI-Native组织）：
> 同样的办公空间但完全不同：只有一个工程师坐在中央，面前是一个大屏幕仪表盘，上面显示多个Agent的工作状态。他身后是发光的AI Agent终端在自动运行。PM和QA的角色被融入Workflow自动流转，墙上不再有手动看板，而是实时数字大屏。整体干净明亮，色调偏蓝白，传达从容高效的感觉。

## 第3页：AI-First 的本质：全流程革命，重新定义人机分工

**开场衔接：**
> 第2页结论：AI革命催生AI-Native的组织形态。但现实是，大多数组织的"AI变革"只是给旧流程贴了一层AI的皮。

**AI-assisted 的现状：**
- 工程师开 Claude Code、产品用 ChatGPT 写 PRD、QA 试试 AI 生成用例
- 流程没变、Sprint 没变、评审没变 — 老流程里塞进新工具
- 效率提升 10-20%，但结构本身没有变化
- 就像织女工换了一把更快的梭子 — 速度变了，生产方式没变

**AI-First：重新定义人机分工：**
- 不是问"AI怎么帮助工程师"，而是 — **如果默认 AI 是主要建设者，流程、架构、组织该怎么重构？**
- 执行者变了：人不再是每个环节的执行者，而是设计者
- 交接方式变了：人传人的信息损耗，变成 Workflow 自动流转
- 对应织女工的类比：不是换梭子，而是从手工织布变成管动力织布机

**瓶颈已经转移：**
- AI 两小时实现一个功能，拖慢团队的不再是开发
- PM 瓶颈：Agent 两小时写完功能，产品规划要几周
- QA 瓶颈：AI 实现完，验证速度跟不上实现速度
- 人头瓶颈：团队规模不可能靠招人追平对手
- 只要有一条链路还是手工的，就会卡住整个系统

**引出第4页：**
> 那全流程革命具体怎么落地？我们需要一套工程实践来回答。

**配图描述（尺寸：800×600px，右侧放置）：**
> 一幅对比示意图。左半边是一个传统软件开发流水线，人（小人图标）在每个环节手动传递文档，流程缓慢、中间有瓶颈（用沙漏图标表示卡顿点）。右半边是AI驱动的全新流水线，AI Agent（发光的机器人图标）在各环节自动执行，人只站在几个关键决策点（用放大镜图标表示），流程流畅无阻。左右之间用一道裂缝隔开，象征范式转变。科技感扁平插画风格。

## 第4页：AI-Native 团队变革：从执行团队到平台团队

**衔接语：**
> 前面讲了，织女工不织布了，去造织布机。程序员也一样 — 不写代码了，去构建让Agent高效运转的平台。

**团队结构变化：**

As-Is（当前）：
```
大多数人在做执行工作 — 写代码、写用例、手动测试、手动部署
少数人在做平台和支持
```

To-Be（未来）：
```
大多数人在做平台工程 — 构建Agent工作流、设计Harness纪律、优化编排效率
Agent负责执行 — 编码、测试、审查、部署
```

> Agent越多、流程越复杂，平台工程团队的比重就越大。未来团队的核心竞争力不是"能写多少代码"，而是"能构建多强的平台让Agent高效运转"。

**能力要求的迁移：**

| 维度 | 当前 | 未来 |
|---|---|---|
| 核心能力 | 编码速度和技术广度 | 意图描述精准度、Agent产出审查判断力 |
| 工作方式 | 自己动手做 | 设计规则、定义标准、构建平台 |
| 价值衡量 | 个人产出量 | 编排效率和平台能力 |

**一句话：** 以前大多数人做执行，少数人做平台；未来大多数人做平台工程，Agent负责执行 — **价值从执行层转移到平台层**。

**配图描述（尺寸：800×900px，左侧竖版放置）：**
> 一个分层的平台架构示意图，从下到上三层：底层是发光的Agent执行层，多个小Agent图标在并行工作（编码、测试、审查、部署），动作快速繁忙；中层是平台层，由工程师构建的蓝色发光基础设施，包括Workflow引擎、Harness纪律、上下文架构等模块；顶层是少数几个工程师站在平台上，俯瞰下方Agent的运作，手里拿着蓝图和调优工具。整张图传达的核心信息是：**工程师不直接做执行工作了，他们站在平台上设计、构建、优化，Agent在下面跑**。科技感分层插画风格，深色背景。

## 第5页：下一代工程师桌面：一个人就是一支团队

**类比：**
- 从前：织女工坐在一台织布机前，一针一线亲手织
- 现在：工厂里十台动力织布机同时运转，她站在车间里巡视、调参数、验质量

工程师的转变一样：
- 从前：一个人坐在编辑器前，一行行亲手写代码
- 现在：你身后站着一群 Agent，编码的、测试的、审查的 — 一个人就是一支全功能团队

**一天的工作变了：**
- 早上到岗，桌面告诉你：昨夜 6 个任务完成、2 个等你确认、1 个需要你的判断
- 你的动作：审查产出 → 给反馈或批准 → 描述新任务 → 回去喝咖啡
- 不再是你追着工作跑，而是**工作做完来找你**

**核心体验：**
- **可见** — 所有进展实时呈现，不需要开会问进度
- **可控** — 关键节点等你拍板，Agent 不会擅自行动
- **可介入** — 方向不对随时喊停、纠正、重来

**一句话：** 以前一个人只能干一个人的活，现在一个人指挥一群 Agent — **你的上限不再是你个人的产能，而是你编排和管理的能力**。

**配图描述（尺寸：1920×800px，顶部横幅放置）：**
> 一个工程师坐在宽大的办公桌前，神态从容放松，手里端着咖啡。他面前的屏幕上是一个简洁的仪表盘界面，显示多个任务卡片在不同状态（进行中、等待确认、已完成）。他身后整齐排列着6-8个半透明的发光AI助手人形，各自在不同工位上忙碌工作（有的在写代码、有的在测试、有的在审查），但都和工程师通过光带连接。整体传达"一个人指挥一群AI"的感觉。未来科技办公风格，明亮干净。

## 第6页：AI-Native桌面的能力底座：感知、编排、观测、控制

**核心内容：**

> 一个人指挥一群Agent干活，桌面必须回答三个问题：Agent怎么知道该干什么、人怎么知道Agent干得怎样、出了问题人怎么管。

**需求感知能力**
- 外部需求自动同步，不用人工搬运
- 需求自动拆解为可执行任务，每个任务携带完整上下文
- Agent 拿到任务就知道做什么、做成什么样，不需要人反复解释

**编排调度能力**
- 任务自动流转 — 编码完进入测试，测试完进入审查，不需要人催进度
- 不同环节交给最合适的 Agent，各展所长
- 多个 Agent 并行执行不同任务，互不干扰

**透明可观测能力**
- Agent 在想什么、调了什么、产出了什么，全程实时可见
- 不是事后看报告，而是像看直播一样看到执行过程
- 卡住了、报错了，第一时间感知

**人在回路能力**
- 关键节点自动暂停等人确认，不擅自行动
- 方向不对随时叫停、纠正、重来
- 审查通过才放行，不通过打回去重做

**一句话：** 感知、编排、观测、控制 — 四种能力让一个人真正管得住一群Agent，**不是放出去不管，而是全程在掌控之中**。

**配图描述：**
> 一个四象限的概念图，中心是一个发光的桌面/屏幕图标。四个象限分别用图标+简短文字表示：左上"感知"（雷达/天线图标，数据流入），右上"编排"（齿轮/管道图标，任务自动流转），左下"观测"（眼睛/监控屏图标，实时透明），右下"控制"（手/开关图标，人做决策）。每个象限之间有箭头连接形成闭环。四象限外围是一圈Agent图标，表示它们被这四种能力统一管理。科技感信息图风格，深色背景配蓝色/青色发光线条。

## 第7页：Coplat：从能力底座到可靠交付的 Harness 实践

**核心内容：**

> Agent 跑一次 demo 很惊艳，但放到生产环境，让 AI 的产出可靠可用，比让它干活难十倍。Coplat 的核心是用 Harness 工程纪律让 Agent 从"能用"变成"可控"。

**四个工程实践：**

**① 上下文架构 — 让 Agent 知道该做什么**
- 不是丢一句"实现这个功能"，而是给 Agent 完整上下文 — 需求描述、验收标准、依赖关系
- 任务源自动同步，任务卡片承载完整信息，Workflow 模板预置上下文传递规则
- 上游步骤的摘要自动传递给下游，Agent 之间不需要人反复解释

**② Agent 专业化 — 让最合适的 Agent 做最合适的事**
- 编码阶段用 Claude Code，验证阶段用 OpenCode，审查阶段用 AI 并行三轮审查
- 不是一个大模型干所有事，而是按环节分配最擅长的执行者

**③ 持久化记忆 — 让每一步都有交付件，每次执行都有积累**
- 每个步骤完成后，执行摘要、产出物、代码变更自动归档
- 执行断了不用重头来 — 从失败步骤重试，上下文链完整保留
- 完整执行过程可回放

**④ 结构化执行 — 给 Agent 划定边界**
- 每个任务在独立 Git Worktree 中执行，多 Agent 并行互不干扰，失败不影响主分支
- 关键节点自动暂停等人确认，确认后跳过已完成步骤继续
- 审查不通过打回重做，通过才放行

**一句话：** Harness 的本质是**给 AI 套上缰绳** — 不是限制能力，而是让能力在可控的轨道上释放。

**配图描述（尺寸：800×900px，左侧竖版放置）：**
> 一个"缰绳"的隐喻视觉：画面中心是一匹充满力量的骏马（代表AI Agent），身上套着精致的蓝色发光缰绳和马鞍（代表Harness工程纪律），缰绳握在一个工程师手中。工程师不是在拉住马，而是轻轻引导方向。马在一条明亮的轨道上奔跑，轨道两侧有护栏和检查站（代表上下文架构、持久化记忆、结构化执行等工程实践）。远处是生产环境的目标。整体传达"不是限制，而是可控地释放力量"。科技感数字插画风格。

## 第8页：两层工作台 — 执行者看板，平台工程搭底座

**核心内容：**

> 第4页讲了团队从执行到平台的转型。落到产品上，就是两层工作台：上层是执行者的看板，下层是平台工程的配置层。

**上层 — AI编排者的工作台（看板区）：**
- 在看板视图管理任务：创建、启动、审查、合并
- 关注任务状态流转：TODO → IN_PROGRESS → DONE
- 在关键节点做决策：审查Agent产出、确认或打回
- 不需要关心Agent怎么配、Workflow怎么编 — 底座已经搭好了

**下层 — 平台工程团队的配置层：**
- Agent配置 — 定义团队成员、分配技能和工具
- Workflow模板设计 — 编排执行步骤、定义上下文传递规则
- Skill开发 — 为Agent注入专业能力
- MCP Server接入 — 连接外部工具和数据源

**一句话：** 执行者只需要关注看板上的任务和决策，底下的Agent、Workflow、Skill、MCP由平台工程团队搭建和维护 — **各司其职，分层解耦**。

**配图描述（尺寸：1920×800px，顶部横幅放置）：**
> 一个清晰的产品分层架构示意图，分为三个水平层级。最上层是"看板区"，展示一个简化的Kanban看板界面缩略图（三列卡片：TODO/IN_PROGRESS/DONE），左侧站着一个工程师图标，标注"AI编排者"，他只看这个界面。中间层是"配置层"，展示四个并排的模块图标：Agent人形图标、Workflow流程线图标、Skill文件图标、MCP Server服务器图标，左侧站着两个工程师图标，标注"平台工程团队"，他们负责构建和维护这些模块。最底层是一个发光的蓝色底座条，标注"Coplat 引擎"。层与层之间用向下的箭头（标注"使用"）和向上的箭头（标注"构建"）连接。整体配色：上层亮白、中层浅蓝、底层深蓝，传达清晰的分层关系。扁平科技插画风格，干净简洁。

## 第9页：实践①上下文架构 — 让Agent拿到任务就知道做什么

**核心内容：**
- 外部需求源（GitHub Issues、企业平台）自动同步为看板任务卡片
- 任务卡片承载完整上下文：需求描述、验收标准、优先级、依赖关系
- Workflow模板预置每一步的指令，上游摘要自动传递给下游
- Agent拿到任务不需要人反复解释，直接执行

**需要截图：**
1. **任务源面板** — Kanban页面中展开Task Source Panel，展示GitHub等外部源同步状态
2. **任务卡片详情** — 一个展开的任务卡片，显示标题、描述、优先级、关联Workflow模板
3. **Workflow模板编辑器** — `/workflow-template` 页面，展示横向步骤卡片，每个步骤有Agent分配和指令

## 第10页：实践②Agent专业化 — 不同环节交给最合适的Agent

**核心内容：**
- Agent配置：定义团队成员，每个Agent绑定技能（Skill）和工具（MCP Server）
- 按环节分配：编码用Claude Code、验证用OpenCode、审查用AI三轮并行
- Agent可测试：通过Chat面板实时对话验证Agent能力
- 不是一个大模型干所有事，而是按环节分配最擅长的执行者

**需要截图：**
1. **Agent配置页** — `/agents` 页面，三栏布局：左侧团队列表、中间Agent详情（技能标签+MCP标签）、右侧Chat面板
2. **Agent Chat面板** — 展开一次对话，显示思考过程和工具调用
3. **Skill配置页** — `/skills` 页面，展示Skill列表和文件浏览器
4. **MCP Server配置页** — `/mcp-servers` 页面，展示服务类型标签和配置

## 第11页：实践③持久化记忆 — 每一步都有交付件，断了不用重头来

**核心内容：**
- Workflow Timeline：完整执行时间线，每个步骤的状态、耗时、产出物一目了然
- 执行过程可回放：Session/Segment/Event三层记录，Agent的每一步操作都有据可查
- 断点重试：执行失败从失败步骤继续，已完成步骤不重复执行
- 模板快照：运行中的Workflow不受模板后续修改影响

**需要截图：**
1. **Workflow Timeline Dialog** — 全屏时间线，展示多个Stage和Node的执行过程，带SVG连接线和进度条
2. **Workflow Progress Dialog** — 960px大弹窗，左侧步骤列表（带状态图标），右侧Session Event日志
3. **InlineWorkflowPanel** — 任务卡片内嵌的紧凑Workflow进度条

## 第12页：实践④结构化执行 — 给Agent划定边界，可并行可控制

**核心内容：**
- Git Worktree隔离：每个任务独立分支和目录，多Agent并行互不干扰
- 看板视图：多任务同时IN_PROGRESS，各自独立推进
- 人在回路：关键节点自动暂停（Suspend），等人确认后继续（Resume）
- Diff审查：Agent产出可直接查看代码变更，确认后合并、不通过打回

**需要截图：**
1. **Kanban看板视图** — `/kanban` 页面，展示TODO/IN_PROGRESS/DONE/BLOCKED四列，多任务并行状态
2. **Diff Viewer** — 代码变更对比弹窗，左侧文件列表，右侧diff内容，带新增/删除统计
3. **Merge Dialog** — 合并弹窗，展示源分支、目标分支、冲突警告
4. **Scheduler面板** — 侧边栏底部调度配置，展示Cron定时和并发控制
