# Coplat AI-Native 落地路径

---

## 一、当前基础

Coplat 已经实现了 AI-Native 的四个核心 Harness 实践：

- **上下文架构**：任务卡片携带完整上下文，上游步骤摘要自动传递给下游，信息不全的任务不允许启动
- **Agent 专业化**：不同步骤绑定不同 Agent，编码、测试、审查各用最合适的执行者
- **持久化记忆**：每步执行摘要和产出物自动归档，断点可重试，执行过程可回放，模板快照保证运行中不受后续修改影响
- **结构化执行**：每个任务在独立 Git Worktree 中执行，多 Agent 并行互不干扰；关键节点暂停等人确认，审查通过才放行

这是 AI-Native 的执行底座。但底座可用和真正能在团队里跑起来，中间还有几个核心挑战要解决。

---

## 二、挑战图谱

AI-Native 的本质是 Agent 替代人做创造性执行。这个替换带来了一系列连锁问题——不只是安全问题，而是从 Agent 的不确定性这个根因出发，衍生出的安全、可靠性、成功率、效率、质量保障五个维度的挑战。

```
Agent 执行的不确定性（根因）
    ├── 安全：Agent 行为可能越界
    ├── 可靠性：Agent 执行结果不稳定
    ├── 成功率：Agent 不是每次都能一次做对
    ├── 效率：Agent 执行有 token 和时间成本
    └── 质量：Agent 产出需要人审，审查是瓶颈
```

### 挑战一：安全 — Agent 行为边界

Agent 在 Worktree 内写代码是安全的。但当 Agent 的能力延伸到 Worktree 之外，风险性质就变了：

- `git push` 把未经审查的代码推到远端
- `gh workflow run` 直接触发 CI/CD 流水线
- `curl` 调用外部 webhook 触发部署
- 修改 `.github/workflows/*.yml` 改变整个团队的交付流程
- `npm publish` / `docker push` 直接发布到生产环境

Coplat 的 Claude Code Executor 以 `--dangerously-skip-permissions` 启动，Agent 有完整的 shell 权限。结构化执行的 Worktree 隔离保护了主分支，但没有保护远端和外部系统。

**需要做的：**

- 工具级别的执行限制：利用 Claude Code 的 `disallowedTools` 和现有 `settingsPath` 机制，为每个 Agent 配置工具权限，禁止高风险命令
- 凭证隔离：Agent 执行环境不注入生产环境的 token 和密钥
- CI/CD 触发权归平台：代码合并、CI 触发、部署发布由平台在人确认后执行，不是 Agent 的 Bash 命令。新增 CI/CD 触发原生步骤类型，走平台调用 API，不走 Agent
- 敏感文件保护：CI 配置文件对 Agent 默认只读

### 挑战二：可靠性 — Agent 执行的稳定性

CI/CD Pipeline 跑的是固定脚本，同样的输入一定产出同样的结果。Agent 不是——同样的 prompt，执行两次可能产出完全不同的代码。这种不确定性在工程上必须被管理。

**具体表现：**

- Agent 中途崩溃或超时，任务卡在半完成状态
- Agent 产出不完整——代码写了一半、测试漏了边界情况
- Agent 执行时间不可控——简单任务 5 分钟，复杂任务可能跑 30 分钟甚至超时
- 网络波动或 API 限流导致 Agent 执行中断

**当前已有的基础：** 持久化记忆保证断点可重试，不需要从头来。但重试本身也有成本。

**需要做的：**

- 执行超时和重试策略：每个步骤设置合理超时，超时后自动重试或暂停等人介入，不是无限等待
- Agent 产出完整性校验：步骤完成后自动检查关键交付物是否存在（比如代码文件是否生成、测试是否执行），不完整的产出不标记为成功
- 执行状态的健康监控：Agent 执行过程中，平台检测到异常（长时间无输出、重复失败）主动告警

### 挑战三：成功率 — Agent 一次做对的比例

Agent 不是每次都能一次通过。成功率受三个因素影响：

1. **上下文质量**：任务描述模糊、验收标准缺失，Agent 就像拿到模糊需求的工程师——做出来的东西大概率不对
2. **任务粒度**：一个大任务拆成三个小任务，每个的成功率 80%，整体成功率只有 51%。拆成更小的任务，整体成功率更高
3. **Agent 能力和任务的匹配度**：让擅长编码的 Agent 做编码、擅长测试的做测试，比一个通用 Agent 做所有事的成功率高

**当前已有的基础：** 上下文架构保证信息完整、Agent 专业化保证技能匹配。但任务粒度的把控目前依赖人。

**需要做的：**

- 任务粒度指引：平台提供任务拆解的建议规则——单次 Agent 执行应该控制在什么范围内，哪些任务应该拆分而不是整体交给 Agent
- 执行结果分类统计：追踪哪些类型的任务成功率高、哪些低，数据驱动优化 Workflow 模板和 Agent 配置
- 从失败中学习：Agent 执行失败后，失败原因（上下文不足、任务过大、Agent 不匹配）自动归类，反哺模板优化

### 挑战四：效率 — Agent 执行的成本

Agent 执行有直接成本（API token 费用）和间接成本（等待时间、重试消耗、审查时间）。

**直接成本：**

- 每个 Agent 步骤消耗 token，一个复杂任务可能消耗数十万 token
- 多个任务并行时，总成本是线性增长的
- 失败重试的成本——重试一次不只是重新跑，还包括平台工程团队排查失败原因的时间

**间接成本：**

- Agent 执行时间不可控，人在等 Agent 完成的过程中是空闲的
- 审查成本——Agent 产出了 500 行代码变更，人要花多长时间审查？

**需要做的：**

- 上下文传递的 token 优化：上游步骤摘要传递给下游时，控制摘要长度，避免把大量原始输出塞进下一步的 prompt
- 并行执行的资源调度：同时运行的 Agent 数量有上限，不是无限制并行。平台需要做队列和调度
- 执行成本可视化：每个任务、每个步骤的 token 消耗和时间消耗可追踪，帮助平台工程团队识别高成本环节

### 挑战五：质量保障 — 审查瓶颈

Agent 两小时写完功能，人要花多长时间审查？如果审查速度跟不上 Agent 的产出速度，Agent 越快，人越累，瓶颈从开发产能转移到审查产能。

**当前已有的基础：** 结构化执行的关键节点暂停、diff 展示。但审查本身仍然依赖人的判断力。

**需要做的：**

- 自动化质量门禁前置：Agent 提交代码后，先跑 lint、类型检查、单元测试，通过后才进入人工审查。把机器能检查的问题拦在人审查之前
- Agent 审 Agent：编码 Agent 的产出先过审查 Agent，审查 Agent 标注风险点和疑问，人只审查 Agent 标注的部分，而不是逐行看 500 行 diff
- 变更风险评估：根据变更范围（文件数、行数、影响模块）自动评估风险等级，低风险变更简化审查流程，高风险变更增加审查环节

## 三、CI/CD 集成

上述五个挑战在 CI/CD 场景下集中爆发——安全风险最大、可靠性要求最高、质量保障最关键。

**CI/CD 集成的 Workflow 设计：**

```
编码步骤（Agent 执行，Worktree 内）
    ↓
测试步骤（Agent 执行，Worktree 内）
    ↓
审查步骤（暂停，人确认产出质量）
    ↓
合并步骤（平台执行 git merge，不经过 Agent）
    ↓
CI 触发步骤（平台调用 CI API，人确认后执行）
    ↓
CI 状态回流（结果同步回 Workflow Timeline）
```

核心原则：**CI/CD 的触发权归平台，不归 Agent。** Agent 的边界是 Worktree 内的代码和测试。代码合并、CI 触发、部署发布由平台在人确认后执行。这不是限制 Agent，而是把"执行"和"交付"的职责划清楚。

## 四、分阶段落地

### 阶段一（近期）

- 为所有 Agent 配置 `disallowedTools` settings 文件，禁止 `git push`、`gh workflow run`、`curl` 等高风险命令——利用现有 `settingsPath`，不需要改代码
- Workflow 模板规范：CI/CD 触发不写进 Agent 的 instructionPrompt
- 步骤超时和重试策略的基础实现
- 自动化门禁前置：lint + 类型检查 + 单元测试在人工审查之前自动执行

### 阶段二（中期）

- Agent 配置页增加工具权限 UI
- CI/CD 触发原生步骤类型，对接 GitHub Actions API
- Agent 产出完整性校验
- 执行结果分类统计，数据驱动优化 Workflow 模板
- Agent 审 Agent（编码 → 审查 Agent 预审 → 人工确认）
- 执行成本可视化（token + 时间）

### 阶段三（长期）

- CI 执行状态回流 Workflow Timeline，完整交付链路可追溯
- 凭证隔离（Agent 环境和 CI/CD 凭证环境分离）
- 敏感文件保护（CI 配置文件对 Agent 只读）
- 并行执行的资源调度和队列管理
- 变更风险评估和分级审查
- 执行异常模式检测和自动告警

## 五、核心立场

四个 Harness 实践（上下文架构、Agent 专业化、持久化记忆、结构化执行）解决的是"Agent 产出质量"的问题。但 AI-Native 要真正落地，还需要解决五个衍生挑战：安全、可靠性、成功率、效率、质量保障。

这五个挑战都源自同一个根因：**Agent 执行的不确定性。** 固定脚本跑错了会报红，Agent 跑错了可能不报错。工程纪律的作用就是把这种不确定性降到可控范围内——不是消除不确定性，而是让不确定性在可接受的范围内运行。

Harness 实践是基础，衍生挑战的治理是进阶。两者加在一起，才是 AI-Native 从"能跑 demo"到"能在团队里长期运行"的完整路径。
