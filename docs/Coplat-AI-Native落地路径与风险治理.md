# Coplat AI-Native 落地路径

---

## 一、当前基础

Coplat 已经实现了 AI-Native 的四个核心 Harness 实践：

- **上下文架构**：任务卡片携带完整上下文，上游步骤摘要自动传递给下游，信息不全的任务不允许启动
- **Agent 专业化**：不同步骤绑定不同 Agent，编码、测试、审查各用最合适的执行者
- **持久化记忆**：每步执行摘要和产出物自动归档，断点可重试，执行过程可回放，模板快照保证运行中不受后续修改影响
- **结构化执行**：每个任务在独立 Git Worktree 中执行，多 Agent 并行互不干扰；关键节点暂停等人确认，审查通过才放行

这是 AI-Native 的执行底座。但底座可用和真正能在团队里跑起来，中间还有四个核心挑战要解决。

---

## 二、挑战图谱

Agent 执行和固定脚本有本质区别：脚本跑错了会报红，Agent 跑错了可能不报错。这种不确定性是所有挑战的根因，衍生出四个维度的问题：

```
Agent 执行的不确定性（根因）
    ├── 安全：Agent 可能做不该做的事
    ├── 可靠性：Agent 执行可能不稳定
    ├── 质量：Agent 产出可能不对
    └── 效率：Agent 执行有资源成本
```

### 挑战一：安全 — Agent 行为边界

Agent 在 Worktree 内写代码是安全的。但 Agent 的能力可以延伸到 Worktree 之外：

- `git push` 把未经审查的代码推到远端
- `gh workflow run` 直接触发 CI/CD 流水线
- `curl` 调用外部 webhook 触发部署
- 修改 `.github/workflows/*.yml` 改变团队交付流程
- `npm publish` / `docker push` 发布到生产环境

Coplat 的 Claude Code Executor 以 `--dangerously-skip-permissions` 启动，Agent 有完整的 shell 权限。结构化执行的 Worktree 隔离保护了主分支，但没有保护远端和外部系统。

**需要做的：**

- 工具级别执行限制：利用 Claude Code 的 `disallowedTools` 和现有 `settingsPath` 机制，为每个 Agent 配置工具权限，禁止高风险命令
- 凭证隔离：Agent 执行环境不注入生产环境的 token 和密钥
- CI/CD 触发权归平台：合并、CI 触发、部署由平台在人确认后执行，不走 Agent。新增 CI/CD 原生步骤类型，走平台调用 API
- 敏感文件保护：CI 配置文件对 Agent 默认只读

### 挑战二：可靠性 — Agent 执行稳定性

固定脚本跑失败了会报错，Agent 跑失败了可能什么都没产出就结束了，或者产出了一半。具体表现：

- Agent 中途崩溃或超时，任务卡在半完成状态
- 产出不完整——代码写了一半、测试漏了边界情况
- 执行时间不可控——简单任务 5 分钟，复杂任务 30 分钟甚至超时
- 网络波动或 API 限流导致执行中断

持久化记忆保证断点可重试，不需要从头来。但重试本身也有成本，频繁重试会拖垮效率。

**需要做的：**

- 超时和重试策略：每个步骤设置合理超时，超时后自动重试或暂停等人介入，不无限等待
- 产出完整性校验：步骤完成后自动检查关键交付物是否存在，不完整的产出不标记为成功
- 执行健康监控：长时间无输出、重复失败等异常状态，平台主动告警

### 挑战三：质量 — Agent 产出正确性

Agent 执行成功了，不代表产出是对的。"看起来对但实际错"是 Agent 产出的典型特征。

影响质量的因素：

- **上下文质量**：任务描述模糊、验收标准缺失，Agent 拿到模糊的需求，做出来的东西大概率不对
- **任务粒度**：一个大任务拆成三个小任务，每个成功率 80%，整体只有 51%。拆细一点，整体成功率更高
- **Agent 和任务的匹配度**：让编码 Agent 做编码、测试 Agent 做测试，比一个通用 Agent 做所有事成功率高

质量保障的另一个问题：**Agent 越快，人越累。** Agent 两小时写完功能，人要花多长时间审查 500 行 diff？审查速度跟不上产出速度，瓶颈从开发产能转移到审查产能。

**需要做的：**

- 自动化门禁前置：lint、类型检查、单元测试在人工审查之前自动跑，机器能查的问题不让人看
- Agent 审 Agent：编码 Agent 的产出先过审查 Agent，标注风险点和疑问，人只看标注的部分
- 任务粒度指引：平台提供拆解建议，控制单次 Agent 执行的范围
- 执行结果分类统计：追踪哪些类型任务成功率高、哪些低，数据驱动优化模板和 Agent 配置
- 变更风险评估：根据变更范围（文件数、行数、影响模块）自动评级，低风险简化审查，高风险增加环节

### 挑战四：效率 — Agent 执行成本

Agent 执行有直接成本和间接成本：

**直接成本：**
- 每个步骤消耗 token，一个复杂任务可能消耗数十万 token
- 多任务并行时总成本线性增长
- 失败重试消耗额外的 token 和时间

**间接成本：**
- 执行时间不可控，人在等待过程中是空闲的
- 审查 500 行 diff 的成本——比 Agent 写代码的时间可能更长

**需要做的：**

- 上下文传递优化：上游摘要传递给下游时控制长度，避免把大量原始输出塞进下一步 prompt
- 并行资源调度：同时运行的 Agent 数量有上限，平台做队列和调度
- 执行成本可视化：每个任务、每个步骤的 token 和时间消耗可追踪，识别高成本环节

## 三、CI/CD 集成

CI/CD 是四个挑战集中爆发的场景——安全风险最大、可靠性要求最高、质量保障最关键。

**Workflow 设计：**

```
编码步骤（Agent 执行，Worktree 内）
    ↓
测试步骤（Agent 执行，Worktree 内）
    ↓
审查步骤（暂停，人确认产出质量）
    ↓
合并步骤（平台执行 git merge，不经过 Agent）
    ↓
CI 触发步骤（平台调用 CI API，人确认后执行）
    ↓
CI 状态回流（结果同步回 Workflow Timeline）
```

核心原则：**Agent 的边界是 Worktree 内的代码和测试。代码合并、CI 触发、部署发布由平台在人确认后执行。** 把"执行"和"交付"的职责划清楚。

## 四、分阶段落地

### 阶段一（近期）

- 为所有 Agent 配置 `disallowedTools` settings 文件，禁止高风险命令——利用现有 `settingsPath`，不改代码
- Workflow 模板规范：CI/CD 触发不写进 Agent 的 instructionPrompt
- 步骤超时和重试策略的基础实现
- 自动化门禁前置：lint + 类型检查 + 单元测试在人工审查前自动执行

### 阶段二（中期）

- Agent 配置页增加工具权限 UI
- CI/CD 触发原生步骤类型，对接 GitHub Actions API
- 产出完整性校验
- Agent 审 Agent（编码 → 审查 Agent 预审 → 人工确认）
- 执行结果分类统计和成本可视化

### 阶段三（长期）

- CI 状态回流 Workflow Timeline，完整交付链路可追溯
- 凭证隔离（Agent 环境和 CI/CD 凭证环境分离）
- 敏感文件保护（CI 配置文件对 Agent 只读）
- 并行执行的资源调度和队列管理
- 变更风险评估和分级审查
- 执行异常检测和自动告警

---

## 五、核心立场

四个 Harness 实践（上下文架构、Agent 专业化、持久化记忆、结构化执行）解决的是"Agent 产出质量"的问题。AI-Native 要真正落地，还要解决四个衍生挑战：安全、可靠性、质量、效率。

这些挑战都源自同一个根因：**Agent 执行的不确定性。** 工程纪律的作用不是消除不确定性，而是让不确定性在可接受的范围内运行。

Harness 实践是基础，衍生挑战的治理是进阶。两者加在一起，才是 AI-Native 从"能跑 demo"到"能在团队里长期运行"的完整路径。
