# AI-Native 组织变革与工程实践

---

## 一、工作模式的变化：DevOps → AI-Native

### 演进路径

```
瀑布 → 敏捷 → DevOps → AI-Native
```

| 阶段 | 解决的问题 | 关键实践 | 执行主体 |
|------|-----------|---------|---------|
| **瀑布** | 规范流程、按阶段交付 | 需求→设计→开发→测试→部署 | 人，串行 |
| **敏捷** | 应对变化、快速迭代 | Sprint、Standup、看板 | 人，小团队迭代 |
| **DevOps** | 打破开发运维壁垒、自动化交付 | CI/CD、IaC、自动化测试、监控 | 人 + 自动化流水线 |
| **AI-Native** | AI Agent 替代人做执行 | 上下文架构、Agent 专业化、持久化记忆、结构化执行 | Agent 执行 + 人审查 |

基线是 **DevOps**。AI-Native 是在 DevOps 基础上的下一步。

### DevOps 做了什么，还差什么

DevOps 已经解决的问题：
- CI/CD 让构建、测试、部署自动化
- IaC 让基础设施可版本管理
- 监控让线上状态可观测
- PR + Code Review 让代码质量有保障

**DevOps 自动化的是"管道"——代码在管道里自动流动，但管道里装的东西（代码、测试、配置）仍然是人写的。**

AI-Native 补的是这一块：**让 Agent 来生产管道里的东西。**

但这里有一个本质区别：CI/CD Pipeline 执行的是确定性脚本（bash、YAML），AI-Native 执行的是 AI Agent。脚本跑错了会报红，Agent 跑错了可能不报红。所以挑战变了——不是怎么编排流程（Workflow 早就有），而是**怎么保证 Agent 的产出质量**。

### DevOps vs AI-Native：三个根本变化

**① 执行者变了：确定性脚本 → AI Agent**

| | DevOps | AI-Native |
|---|---|---|
| 执行者 | bash 脚本、YAML Pipeline | AI Agent |
| 确定性 | 输入确定 → 输出确定 | 同样输入可能产出不同结果 |
| 上下文依赖 | 不需要，脚本按固定逻辑跑 | 强依赖，信息不完整产出就是错的 |
| 失败模式 | 报错，日志可查 | 可能不报错但产出是错的 |
| 管理重点 | Pipeline 规范、测试覆盖率 | 上下文完整性、产出审查、门禁 |

核心问题：DevOps 管的是程序，错了能查到。AI-Native 管的是 Agent，**它可能不报错但做错了**。

**② 质量保障变了：事后检查 → 过程中约束**

| | DevOps | AI-Native |
|---|---|---|
| 代码质量 | PR Review + Lint | Lint + 门禁 + 人做最终审查 |
| 测试 | 人写用例，CI 跑 | Agent 生成用例 + 自动执行 + 覆盖率门禁 |
| 安全 | SAST/DAST 扫描 | 门禁嵌入流程，不允许引入风险依赖 |
| 发布 | Feature Flag、灰度、回滚 | 关键节点人确认 + 变更可追溯可回滚 |
| 隔离 | 分支策略 | 每个 Agent 在独立 Worktree 执行 |

DevOps 的质量检查在"事后"——东西做完了再跑 Pipeline。AI-Native 的质量约束必须在"过程中"——Agent 执行的每一步都有门禁，因为 Agent 产出错误的代价远大于脚本报错。

**③ 管理对象变了：管人 → 管 Agent 产出**

| | DevOps | AI-Native |
|---|---|---|
| 规划周期 | 两周一个 Sprint | 按任务，不按 Sprint |
| 进度跟踪 | 看板 + Burn-down 图 | 实时仪表盘 |
| 管理对象 | 人——谁在做、做了多少 | 任务——什么状态、产出质量 |
| 会议作用 | Standup 同步进度 | 做关键决策 |
| 瓶颈位置 | 开发产能（人不够） | 需求精度和审查速度（人审不过来） |
| 协作边界 | 按职能划分 | 按业务域划分（一个人 + 多个Agent） |
| 扩展方式 | 加人 | 加 Agent |

### AI-Native 的四个关键实践

Workflow 不是新东西。AI-Native 的关键实践也不是 Workflow 本身，而是**因为执行者变成了 AI Agent，必须建立新的工程纪律**：

**实践一：上下文架构 — 让 Agent 知道该做什么**

脚本不需要理解需求，Agent 需要。给 Agent 的信息不完整，它不会报错，会直接产出错误结果。所以必须：
- 任务卡片携带完整上下文（需求描述、验收标准、依赖关系）
- 上游步骤的执行摘要自动传递给下游
- 信息不全的任务不允许启动

**实践二：Agent 专业化 — 不同环节用不同的 Agent**

CI/CD Pipeline 的每个 stage 是固定脚本，不需要考虑"谁擅长什么"。Agent 不同，不同模型擅长的领域不一样：
- 编码阶段用擅长写代码的 Agent
- 测试阶段用擅长测试的 Agent
- 审查阶段用擅长 Review 的 Agent
- 按环节分配最合适的执行者，不是一个 Agent 干所有事

**实践三：持久化记忆 — 每一步都有交付件，断了不用重头来**

CI/CD Pipeline 的每个 stage 失败了可以重跑，但状态不会保留。Agent 执行不同——一个多步任务跑到第三步失败了，不应该从第一步重来。所以必须：
- 每步完成后，执行摘要、产出物、代码变更自动归档
- 执行失败从失败步骤重试，已完成步骤不重复执行
- 完整执行过程可回放，出问题可追溯
- 运行中的 Workflow 不受模板后续修改影响（模板快照）

**实践四：结构化执行 — 给 Agent 划定边界，可并行可控制**

CI/CD Pipeline 是单线串行的，不存在并行冲突问题。Agent 执行不同——多个 Agent 同时跑不同任务，需要隔离和控制：
- 每个任务在独立 Git Worktree 中执行，多 Agent 并行互不干扰，失败不影响主分支
- 关键节点自动暂停等人确认，确认后跳过已完成步骤继续
- 审查不通过打回重做，通过才放行
- Agent 产出可直接查看代码变更 diff，确认后合并

### 误区

AI-Native 不是替代 DevOps，而是在 DevOps 基础上叠加：
- CI/CD 仍然需要，但触发者从人变成 Agent
- 代码审查仍然需要，但第一轮从人变成 AI
- 监控仍然需要，但监控对象从服务变成 Agent 行为
- 测试仍然需要，但用例从人写变成 Agent 生成

**没有 DevOps 成熟度的团队直接上 AI-Native，会出问题。**

## 二、团队结构：谁会缩减，谁是关键角色

### 会缩减或消失的角色

| 角色 | 现状 | 未来 | 原因 |
|------|------|------|------|
| 纯执行型开发者 | 团队主体 | 大幅缩减 | Agent 写代码比人快、比人稳定 |
| 手动测试工程师 | QA 团队核心 | 大幅缩减 | AI 生成用例 + 自动执行，手动测试只剩探索性测试 |
| 项目协调者 | 必不可少 | 消失 | 进度实时可见，不需要人催进度 |
| 初级运维 | 运维团队基础 | 缩减 | 部署、环境配置等标准化操作 Agent 可完成 |

### 不会消失但会剧变的角色

| 角色 | 现在做什么 | 未来做什么 |
|------|-----------|-----------|
| 产品经理 | 写 PRD、排优先级 | 定义精确的验收标准（模糊的代价被极速放大） |
| 架构师 | 画架构图、做技术选型 | 设计 Agent 的上下文架构和执行编排 |
| 技术负责人 | 管人、盯进度 | 管 Agent 产出质量和平台工程纪律 |

### 新增的关键角色

| 角色 | 核心职责 | 为什么关键 |
|------|---------|-----------|
| **AI 编排者** | 设计执行流程、审查 Agent 产出、做关键决策 | 一个人管多个 Agent 的产出 |
| **平台工程师** | 构建 Agent 运行的基础设施——执行引擎、上下文架构、门禁规则 | Agent 越多，平台越复杂，这个角色越不可替代 |
| **上下文架构师** | 设计信息在 Agent 之间怎么流转，保证每个 Agent 拿到的信息完整 | 信息不完整，Agent 产出就是错的 |
| **Agent 运维** | 监控 Agent 行为、处理异常、优化执行效率 | Agent 不完美，需要人兜底 |

### 团队阵型变化

从"金字塔"到"菱形"——大量执行者在底层的结构消失，平台工程师成为最厚的中间层。

```
传统：                          未来：

     △ 管理层                    △ 决策层
    ╱  ╲                        ╱  ╲
   ╱    ╲                      ╱ 平台 ╲
  ╱ 执行层 ╲                   ╱ 工程  ╲
 ╱──────────╲                  ╱──────────╲
  大量执行者                    Agent 执行层
```

## 三、团队规模与比例

### 现状：20 人团队的典型结构

```
开发 14人 (70%)  ████████████████████
QA    3人 (15%)  ██████
PM    2人 (10%)  ████
DevOps 1人 (5%)  ██
```

70% 的人在做执行。平台/支撑是极少数。

### 未来：同等产出的 AI-Native 团队

**8-10 人 + 15-25 个 Agent**

```
平台工程 4-5人 (~50%)
  · 执行流程设计、Agent 配置、门禁规则
  · Skill 开发、MCP Server 接入
  · 上下文架构维护

AI 编排 1-2人 (~15%)
  · 审查 Agent 产出、做关键决策
  · 处理 Agent 解决不了的异常

产品/需求 1-2人 (~15%)
  · 定义精确的验收标准
  · 拆解需求为可执行任务卡片

Agent 运维 1人 (~10%)
  · 监控 Agent 行为、处理故障
  · 调优并发、优化执行效率

─────── Agent 层（不计入人力）──────
Agent 15-25个
  · 编码 × 5-8 · 测试 × 3-5
  · 审查 × 2-3 · 部署 × 2-3
  · 其他专用 × 3-6
```

### 核心比例变化

| 维度 | 传统团队 | AI-Native 团队 |
|------|---------|---------------|
| **执行 : 平台** | 7 : 1 | **1 : 4** |
| **人力 : Agent** | 20 : 0 | **10 : 20** |
| **管理方式** | 1 个 TL 管 7 个开发 | 1 个人管 8-10 个 Agent |
| **人均产出** | 每天约 0.2 个功能 | 每天约 2-5 个功能（含 Agent 产出） |

执行和平台的比例从 7:1 翻到了 1:4。

### 按规模推算

| 规模 | 传统人力 | AI-Native 人力 | Agent 数量 | 人力缩减 |
|------|---------|---------------|-----------|---------|
| 小队（5-8人） | 6人 | **3-4人** | 5-8个 | ~40% |
| 中队（20人） | 20人 | **8-10人** | 15-25个 | ~50% |
| 大组（100人） | 100人 | **25-35人** | 80-150个 | ~70% |

规模越大收益越高——平台工程有规模效应，10 个人搭的平台可以支撑 50 个 Agent。

### 关键前提

这个结构成立的前提：**平台工程的那 4-5 个人必须强。** 他们不是写业务代码，而是构建 Agent 运行的工程纪律：

- 上下文架构不好 → Agent 信息不完整 → 产出错误
- 执行流程不好 → 步骤衔接断裂 → 频繁人工介入
- 门禁设置不好 → 要么卡死流程，要么 Agent 产出失控

**真正稀缺的是平台工程师——设计这套系统需要深厚的工程功底。**

## 四、治理体系

### A. 管理层（靠人判断）

| 治理项 | 为什么需要人 | 怎么做 |
|--------|------------|--------|
| **需求质量** | 需求不精确 = Agent 产出错误，这是源头 | PM 提交需求前必须定义可执行的验收标准 checklist |
| **产出审查** | Agent 写的代码必须有人审查后才能合并 | 编排者在关键节点审查，通过才放行 |
| **方向校准** | Agent 执行偏离预期时，人判断是调整还是叫停 | 执行流程暂停，人决策 |

### B. 工程治理（靠平台自动化）

| 治理项 | 怎么做 | 为什么 |
|--------|-------|-------|
| **上下文完整性** | 任务信息不全不允许启动 | 信息不完整，产出必错 |
| **Agent 专业化** | 不同任务绑定不同 Agent，不允许乱分配 | 编码任务不该用测试 Agent 跑 |
| **持久化** | 每步执行必须有摘要和产出物归档 | 断了能从断点恢复，过程可追溯 |
| **隔离** | 每个任务在独立 Worktree 执行，不碰主分支 | 多 Agent 并行互不干扰 |

### C. 门禁（自动化 + 人工确认）

| 门禁 | 检查内容 | 触发时机 |
|------|---------|---------|
| **代码质量** | lint 通过、覆盖率达标、安全扫描无高危 | Agent 提交代码后自动触发 |
| **人工确认** | 人在关键节点确认后才放行 | 合并主分支、发布生产环境 |
| **安全** | 不引入已知漏洞依赖、不硬编码密钥 | 每次代码变更自动检查 |
| **回滚** | 每次变更必须有可回滚路径，合并前展示完整 diff | 合并前强制 |
| **并发** | 同时运行的 Agent 数量有上限 | 执行启动前检查 |

### 治理原则

**人管方向和质量判断，平台管流程和纪律，门禁管安全和底线。**

```
┌──────────────────────────────────────┐
│         门禁（底线）                  │
│   安全 · 回滚 · 质量 · 并发          │
│  ┌────────────────────────────────┐  │
│  │       工程治理（流程）          │  │
│  │  上下文 · Agent分配 · 持久化    │  │
│  │  ┌────────────────────────┐    │  │
│  │  │    管理层（人的判断）    │    │  │
│  │  │  需求质量 · 产出审查    │    │  │
│  │  └────────────────────────┘    │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

## 五、什么样的团队适合做

### 适合的

| 条件 | 为什么 |
|------|-------|
| **工程成熟度高** | 已有 CI/CD、代码审查、自动化测试。这些是基础，没有这些管不住 Agent |
| **重复性高的业务** | CRUD、API 开发、表单页面、数据迁移、测试用例生成。任务定义清晰、验收标准明确 |
| **有平台工程基础** | 已有人在 DevOps 工具链、内部开发平台。转 Agent 平台是自然延伸 |
| **需求可结构化** | 需求能拆解为任务卡片，验收标准能写 checklist |

### 不适合或暂时不适合的

| 场景 | 原因 |
|------|------|
| **0→1 探索型产品** | 需求还在变化，验收标准无法定义，Agent 接不到有效指令 |
| **工程纪律薄弱** | 没有代码审查、没有自动化测试。给 Agent 只会加速混乱 |
| **重度领域知识** | 嵌入式、芯片设计、医疗合规——领域知识难以结构化传递 |
| **极小团队（<3人）** | 搭平台的成本高于直接用 AI 工具辅助个人效率 |

### 三个阶段

```
阶段一：AI-Assisted（大多数人在这里）
├── 个人用 AI 工具提效
├── 流程不变
├── 效率提升 10-20%
└── 门槛：个人愿意学新工具

        ↓ 门槛：工程纪律

阶段二：AI-First（成熟团队正在进入）
├── 流程围绕 AI 重构
├── 人从执行者变成审查者
├── 效率提升 3-5x
└── 门槛：团队工程成熟度

        ↓ 门槛：组织变革的决心

阶段三：AI-Native（少数团队在探索）
├── Agent 做执行，人做平台和决策
├── 平台工程是核心竞争力
├── 效率提升 10x+
└── 门槛：组织重构 + 平台工程投入
```

大部分团队卡在阶段一，不是技术问题，是管理惯性问题。
