# Coplat 当前能力与下一步工作重点

> 基于代码库实际实现情况，校准能力全景图，明确下一阶段优先级。

---

## 一、平台定位

Coplat 是一个 AI-Native 的研发看板平台。核心理念是 **Harness Engineering** — 为 AI Agent 构建工程纪律，让 Agent 从"能跑 demo"变成"能在团队中长期可靠运行"。

平台围绕四个核心实践构建：

| 实践 | 解决的问题 |
|------|-----------|
| **上下文架构** | Agent 拿到任务就知道做什么，不需要人反复解释 |
| **Agent 专业化** | 不同环节交给最合适的 Agent，不是一个大模型干所有事 |
| **持久化记忆** | 每步有交付件，断点可重试，执行可回放 |
| **结构化执行** | Worktree 隔离、人在回路、关键节点可控 |

## 四、未实现能力与优先级

按**对核心流程的影响程度**和**实现难度**排列优先级。

### P0 — 核心流程补全（必须做）

这些是"Agent 从写完代码到交付上线"链路中的关键缺失环节。

| 能力 | 所属维度 | 当前痛点 | 预估工作量 |
|------|---------|---------|-----------|
| **CI/CD 流水线对接** | 结构化执行 | Agent 写完代码后无法自动触发构建和测试，手动操作打断自动化链路 | 大 |
| **自动化测试执行** | Agent 专业化 | Agent 产出的代码没有自动化验证，全靠人看 | 中 |
| **AI 三轮并行审查** | Agent 专业化 | 代码审查完全依赖人，审查速度跟不上 Agent 产出速度 | 大 |
| **质量门禁** | 结构化执行 | 无自动化的 lint/类型检查/安全扫描，问题流入人工审查 | 中 |

**为什么是 P0：** 当前 Coplat 覆盖了"需求进来 → Agent 写代码"的链路，但"写完代码 → 验证质量 → 审查通过 → 合并交付"这后半段几乎全靠人手动操作。后半段的自动化程度直接决定了平台能承接的任务量和团队信任度。

### P1 — 可靠性与成本控制（应该做）

| 能力 | 所属维度 | 当前痛点 | 预估工作量 |
|------|---------|---------|-----------|
| **Token 成本追踪** | 效率 | 不知道每个任务/步骤消耗了多少 token，无法识别高成本环节 | 中 |
| **执行超时与重试策略** | 结构化执行 | Agent 可能长时间卡住无输出，无超时机制 | 中 |
| **异常状态主动告警** | 监控 | Agent 卡住或失败后，编排者不能第一时间感知 | 中 |
| **工具权限控制 (disallowedTools)** | 安全 | Agent 以 `--dangerously-skip-permissions` 启动，无工具级别限制 | 小 |
| **步骤产出物自动校验** | 持久化记忆 | 步骤完成后不检查交付物是否存在，不完整的产出被标记为成功 | 中 |

### P2 — 体验与效率提升（可以做）

| **执行结果分类统计** | 监控 | 哪些任务成功率高/低，驱动优化模板和 Agent 配置 | 中 |

## 五、下一阶段工作重点

### 阶段一：利用现有机制，补全安全与监控基础

**目标：** 不改核心架构，通过配置和轻量改动，让当前流程更安全、更透明。

| 工作项 | 做法 | 工作量 |
|--------|------|--------|
| 工具权限控制 | 利用现有 `settingsPath` 机制，为每个 Agent 配置 `disallowedTools`，禁止高风险命令 | 小 |
| Workflow 模板规范 | CI/CD 触发不写进 instructionPrompt，明确 Agent 职责边界 | 文档规范 |
| 执行超时机制 | 步骤级别配置超时阈值，超时自动标记为 FAILED 并通知 | 中 |
| Token 消耗追踪 | 解析 Agent 输出中的 token 用量，记录到 Session Event，前端展示 | 中 |
| 异常状态告警 | 超时无输出、重复失败等状态，通过现有 Webhook 通知推送 | 中 |

### 阶段二：补全交付链路，建立质量门禁

**目标：** 打通"代码写完 → 自动验证 → 审查 → 合并"的完整链路。

| 工作项 | 做法 | 工作量 |
|--------|------|--------|
| CI/CD 原生步骤类型 | 新增 CI/CD 步骤类型，Workflow 执行到该步骤时由平台调用 CI API（如 GitHub Actions），不走 Agent | 大 |
| 自动化测试执行 | 在编码步骤后新增测试步骤，Agent 自动运行项目测试并收集结果 | 中 |
| 质量门禁 | lint + 类型检查 + 单元测试在 Agent 提交后自动执行，不通过则标记失败 | 中 |
| AI 三轮并行审查 | 编码产出先经过三个并行审查 Agent（质量/安全/依赖），标注风险点，人只看标注部分 | 大 |
| 步骤产出物校验 | 步骤完成后自动检查关键交付物（如文件是否存在、diff 是否为空） | 中 |
| 产出完整性与成本可视化 | 前端展示每个任务的 token 消耗、耗时、步骤级明细 | 中 |

### 阶段三：完整治理闭环

**目标：** 平台具备长期稳定运行的治理能力。

| 工作项 | 做法 |
|--------|------|
| CI 状态回流 Timeline | CI 结果自动更新到 Workflow Timeline，完整链路可追溯 |
| 凭证隔离 | Agent 执行环境与 CI/CD 凭证环境分离 |
| 变更风险评估 | 根据变更范围（文件数、行数、影响模块）自动评级，分级审查 |
| Agent 记忆系统 | 跨任务积累经验，避免重复错误 |
| 自愈闭环探索 | 监控 → AI 分诊 → 自动工单 → 修复 Workflow → 验证 |

## 六、核心竞争力总结

当前 Coplat 已经建立的核心壁垒：

1. **Workflow 引擎 + 多执行器** — 不是单次 Agent 调用，而是多步骤、多 Agent 的工程化编排
2. **Worktree 隔离 + Suspend/Resume** — 多任务并行安全、关键节点人在回路
3. **Session/Segment/Event 三层记录 + Time Travel 重试** — 完整可追溯、断点可恢复
4. **6 种任务源适配器 + 自动调度** — 需求进来全链路自动运转
5. **Skill + MCP 按步骤注入** — 每步给 Agent 精准的能力，而非全量工具

**下一阶段的核心命题：** 把"Agent 写完代码"到"代码合入主分支"这条链路自动化、工程化。当前后半段几乎全靠人，这是平台能否真正承担团队日常交付的关键门槛。
