## 详细表格

### 第一层：工作流阶段

| # | 阶段 | 说明 |
|---|------|------|
| 1 | **需求接纳** | 从外部系统或手动创建需求，进行需求澄清、分解和排期 |
| 2 | **设计分析** | 对需求进行技术可行性分析、架构设计、任务拆解 |
| 3 | **代码开发** | AI Agent 在隔离环境中执行编码任务 |
| 4 | **开发自测** | 代码变更的验证、测试执行和质量检查 |
| 5 | **代码审查** | 变更审查、合并决策、冲突解决 |
| 6 | **漏洞修复** | CVE 漏洞识别、影响评估、升级修复 |
| 7 | **部署发布** | 代码合并主干、版本发布、部署上线 |

## 第二层：场景 + 输入/输出

### 1. 需求接纳

| 场景 | 描述 | 输入 | 输出 | 状态 |
|------|------|------|------|------|
| **外部需求同步** | 从 GitHub Issues / CloudDevOps 等外部系统批量拉取需求 | 外部系统配置（repo、token、API 地址） | 标准化任务卡片（标题、描述、标签、外部链接） | `[已实现]` |
| **手动创建需求** | 用户在 Kanban 面板直接创建任务/需求 | 任务标题、描述、优先级、指派人 | 创建的任务卡片，进入 TODO 列 | `[已实现]` |
| **需求澄清** | 通过与 AI 对话澄清模糊的需求 | 需求描述 | 明确的需求说明、验收标准 | `[待实现]` |
| **需求分解** | 将大型需求/Epic 拆解为可执行的任务子项 | 需求描述、上下文 | 子任务列表，含标题、描述、依赖关系 | `[待实现]` |
| **需求排期** | 将需求分配到迭代(Sprint)，设置优先级和时间范围 | 需求列表、迭代信息、团队容量 | 排期计划，任务归属迭代 | `[已实现]` |

### 2. 设计分析

| 场景 | 描述 | 输入 | 输出 | 状态 |
|------|------|------|------|------|
| **代码仓库探索** | AI Agent 自动分析代码仓库结构和技术栈 | Git 仓库地址 / 本地目录路径 | 仓库结构报告、技术栈分析、关键模块说明 | `[已实现]` |
| **技术方案生成** | AI 根据需求生成技术实现方案 | 需求描述、仓库分析结果 | 技术方案文档，含架构设计、接口定义、实现步骤 | `[待实现]` |

## 第四层：竞争力与爆点总结

### 核心竞争力

| # | 竞争力 | 说明 |
|---|--------|------|
| 1 | **AI 原生的任务执行** | 不是"AI 辅助看板"，而是"AI 驱动开发"。任务不只被管理，更被自动执行——从需求到代码到验证的完整闭环 |
| 2 | **多 Agent 编排引擎** | 基于 Mastra 的工作流引擎支持多步骤、多 Agent 串联/并行执行，每个步骤可绑定不同角色的 Agent（架构师→前端→后端→QA），实现真正的"虚拟开发团队" |
| 3 | **Git Worktree 原生隔离** | 每个任务自动创建独立 Git Worktree 和 feature 分支，实现真正的任务级代码隔离——多任务可并行开发互不干扰，这是传统看板工具做不到的 |
| 4 | **人机协同中断机制** | `requiresConfirmation` + `AskUserQuestion` 双重中断点，AI 不是黑盒执行，关键决策点由人把控，实现"AI 干活、人决策"的协作模式 |
| 5 | **可插拔的技能与工具生态** | Skill 系统 + MCP Server 管理让 Agent 能力可动态扩展；Bundle 打包让配置可复用可分发 |

### 差异化爆点

| # | 爆点 | 亮点 |
|---|------|------|
| 1 | **工作流即代码** | 工作流模板可导入导出、可版本管理、可团队共享。一个精心调优的"漏洞修复工作流"可以直接分发给整个团队 |
| 2 | **CVE 漏洞一键修复** | 内置 5 步标准化漏洞修复工作流（分析→评估→方案→执行→验证），从收到 CVE 告警到修复验证全自动化，大幅缩短安全响应时间 |
| 3 | **外部系统无缝对接** | 插件化 Task Source 适配器架构，GitHub Issues / CloudDevops 等外部系统的需求可直接同步为任务，打通"需求来源 → AI 执行 → 代码交付"的全链路 |
| 4 | **Time Travel 重试** | 工作流失败时使用 `timeTravelStream` 从失败步骤重试，不重新执行已完成的步骤——省时、省钱、省 Token |
| 5 | **全链路可视化** | 从看板视图到工作流时间线到实时事件流，AI 执行过程完全透明可观测——消息、工具调用、思考链、耗时、Token 消耗一览无余 |
