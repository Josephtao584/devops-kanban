1. 修正后的核心观点

session 没问题，问题是 session 表里塞了太多不属于 session 本体的字段。所以需要做“会话本体”和“归属/编排/视图投影”分离。

Session 一次独立 agent 会话 负责：身份、状态、基础上下文
+
SessionEvent 消息 / tool / stdout / stderr / status append-only 历史
+
SessionOwner / Projection 谁拥有它、在哪展示、如何轮询 这些不进 session 本体

2. 什么叫“不纯”

不纯不是 session 这个词不对，而是一个实体混了多个层次的职责。

字段类别 是否应该进 session 原因
session 自身身份
id / status / created_at / updated_at		 应该 这是会话本体属性
执行上下文
agent_id / executor_type / worktree_path / branch / initial_prompt		 应该 这是会话运行所需的基础上下文
owner 归属
workflow_step_id / workflow_run_id / task_id		 谨慎 可以有，但最好是关联层或 owner 映射，不要把 session 变成 orchestration 总表
编排投影
current_step / workflow summary / ui selection		 不应该 这是 workflow 或 view model 的职责
传输语义
websocket channel / polling cursor		 不应该 这是读取协议/应用层职责，不属于领域会话
大块历史输出
完整 output string 作为唯一真相		 不应该 历史应由 event log 主导，output 只能做兼容快照或摘要

3. 推荐的纯化拆分

核心思路：session 保留，但拆掉多余职责。

sessions

  • id
  • status
  • agent_id
  • executor_type
  • worktree_path
  • branch
  • initial_prompt
  • started_at / completed_at

session_events

  • session_id
  • seq
  • kind
  • role
  • content
  • payload
  • created_at

workflow_step_session_links

  • workflow_run_id
  • step_id
  • session_id
  • display summary
  • error
  • started_at / completed_at
如果你不想单独建 link 表，也可以把 session_id 轻量放在 workflow run step 摘要里，但不要把 workflow 编排字段反向塞满 session。

4. session 本体建议字段

这一版更聚焦“会话自身”。owner 关系和视图投影只保留最小必要引用。

字段 是否建议保留 说明
id保留会话标识
status保留CREATED / RUNNING / COMPLETED / FAILED / STOPPED
agent_id保留运行该会话的 agent
executor_type保留决定如何解释执行结果
worktree_path保留运行上下文
branch保留运行上下文
initial_prompt保留会话起始输入
task_id可保留如果 session 天然属于某任务，可接受
workflow_run_id尽量不主挂除非查询性能强依赖，否则更建议放 link/projection
workflow_step_id尽量不主挂更像 owner 关系，不是 session 自身属性
output仅兼容保留不能再作为唯一真相，权威来源应是 session_events
poll_after_seq / ws_channel删除属于传输层，不属于领域模型
重点：session 可以是核心对象，但要尽量只表达“会话是什么”，不要表达“workflow 怎么编排它”“前端怎么读它”“传输层怎么订阅它”。

5. 这版 workflow 怎么接 session

workflow step 与 session 关联，但不反向污染 session 本体。

WorkflowRun.step 状态摘要 session_id（轻量引用）
→
Session 这一步的独立会话 负责事件历史与运行状态
→
SessionEvents 右侧聊天窗口真实来源 按 seq 增量轮询
这样 workflow 只知道“这个 step 对应哪个 session”，不会要求 session 反向承载整套 workflow 编排语义。

6. 前端读取方式

第一版不做 WebSocket，点击 step 后直接走 session 详情 + event 增量轮询。

7. 当前修正版结论

我现在理解你的意思了：session 保留，但 session 字段要纯化，不能混成一个“万能执行上下文大对象”。

这版应该坚持

  • workflow step = 独立 session
  • session history = session_events
  • step 右侧面板读 session + events
  • 实时靠轮询 after_seq

这版要避免

  • 把 workflow 编排状态塞进 session
  • 把传输层字段塞进 session
  • 把 output string 当唯一来源
  • 让 session record 承担 owner projection
如果你认可这次修正，我下一步就按这个方向把字段级设计继续收紧：session 字段白名单、session_events 结构、workflow step 到 session 的最小关联方式。
