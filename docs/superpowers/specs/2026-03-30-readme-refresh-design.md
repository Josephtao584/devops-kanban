# README 更新设计

## 背景
当前 `README.md` 已经较完整地表达了项目的 Harness Engineering 定位，但存在两个问题：

1. 信息密度高，首次访问者很难快速抓住项目是什么、怎么启动、怎么使用。
2. 部分内容与当前仓库结构和近期演进不完全同步，尤其是 workflow/skill 相关能力、目录说明与功能重心。

本次更新的目标不是重塑品牌叙事，也不是扩写成完整产品文档，而是在保留中文定位和核心理念的前提下，对 README 做结构重整和内容校准。

## 目标
- 保留中文 README 和 Harness Engineering 主线表达。
- 提升首次访问者的理解效率，让用户能快速知道“这是什么”“怎么跑”“怎么用”。
- 让 README 与当前代码库状态更加一致。
- 控制篇幅，避免 README 承担过多设计文档职责。

## 非目标
- 不改成中英双语。
- 不新增独立文档体系。
- 不把 README 扩写成完整的 API 手册或任务源开发指南。
- 不修改除 `README.md` 之外的项目文档。

## 读者假设
README 的主要读者分为三类：

1. **首次访问仓库的开发者**：需要快速理解项目定位和启动方式。
2. **准备本地试用的贡献者**：需要准确的运行命令、目录结构和开发入口。
3. **关注 agent workflow / harness engineering 的产品或平台开发者**：需要看到项目的核心设计思路，但不需要在 README 中阅读所有实现细节。

## 更新策略
采用“结构重整型更新”：

- 保留现有 README 中已经形成特色的中文表达与 Harness Engineering 定位。
- 压缩重复解释理念的段落，把高频查阅的信息前置。
- 修正与当前仓库不一致的功能说明、目录结构、命令与架构描述。
- 适度补充 workflow 系统与当前实现重点，但不展开到底层设计细节。

## README 目标结构

### 1. 项目简介
用简洁的开头回答三个问题：
- 这是什么项目
- 它解决什么问题
- 它适合谁

这一部分保留项目的定位表达，但篇幅显著短于现有版本，避免在开篇进入长段理念论述。

### 2. 核心能力
集中列出当前项目最关键的能力，包括：
- 项目 / 需求 / 任务看板管理
- AI Agent 配置与执行
- Git Worktree 隔离执行
- Session / Execution / WebSocket 可观察性与可恢复性
- Workflow 模板与执行体系

此处重点是“让读者知道系统能做什么”，而不是展开完整原理解释。

### 3. 快速开始
将启动说明前置，并确保与当前仓库一致：
- Node.js 版本要求
- `./start.sh` 一键启动
- 前后端手动启动方式
- 默认访问地址

命令以当前 `CLAUDE.md` 和仓库实际约定为准。

### 4. 典型使用流程
用一个简短、面向用户操作的流程说明系统如何被使用：
- 创建项目
- 配置 Agent
- 创建任务 / 需求
- 启动执行
- 执行中交互
- 审查与合并

这一部分强调 human-in-the-loop，而不是把平台描述成全自动系统。

### 5. 架构概览
简洁说明：
- 后端分层结构（routes / services / repositories）
- workflow 子系统（workflowService / lifecycle / executors / repository serialization）
- 前端结构（views / components / stores / api）
- `data/` JSON 文件存储

目标是给开发者一个可信的当前架构快照。

### 6. 开发说明
聚合开发者常查信息：
- 常用命令
- 常见注意事项（如响应格式、数据目录、端口）
- 可扩展点（任务源、执行器 / Agent）

这里会保留扩展方向，但不再在 README 中长篇展开 `config.yaml`/JSONPath/transforms 等细节实现。

### 7. 贡献与许可
保留贡献入口和 License，简化表达。

## 内容取舍规则

### 保留
- Harness Engineering 的项目定位
- 隔离执行、可观察性、可恢复性、人机协作这些项目核心思想
- 快速开始、使用流程、架构概览这些基础信息

### 压缩
- 多段重复解释“为什么不是普通 AI demo”的内容
- 过长的理念性段落
- 对 README 主流程帮助不大的冗长示例

### 移除或弱化
- 过细的通用任务源配置细节
- 容易随代码变化而过时的过度展开说明
- 与当前项目状态不一致的描述

## 风格要求
- 中文为主
- 保持 GitHub 首页文档风格
- 用短段落、列表、表格替代冗长论述
- 尽量先回答“用户最关心的问题”，再展开理念

## 验收标准
README 更新完成后，应满足以下标准：

1. 新读者在开头部分即可理解项目定位与核心价值。
2. 开发者无需阅读整篇即可完成本地启动。
3. README 中的架构和命令与当前仓库结构基本一致。
4. 文档保留项目独特定位，但不显得冗长重复。
5. 修改范围限定在 `README.md`。

## 实施边界
本次只更新 `README.md` 文本内容，不涉及：
- 代码变更
- 新增截图、图片资源
- API 文档生成
- 其他文档文件的同步改写
