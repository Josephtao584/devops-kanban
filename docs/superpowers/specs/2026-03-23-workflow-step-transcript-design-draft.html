Workflow Step Transcript Design Draft

目标：workflow 跑起来后，点击某个 step，可以在右侧看到这个 step 的完整历史与持续追加的结构化对话流。第一版 不使用 WebSocket，通过持久化 + 增量轮询实现实时感。

No WebSocket
Domain First
Polling for Incremental Reads
Claude First UI Optimization
Executor-agnostic Core Model

1. 核心领域模型

把模型从 execution session 改成更纯的运行/记录模型：WorkflowRun → StepRun → Transcript → TranscriptEvent。

WorkflowRun 一次工作流运行 只保留编排摘要
→
StepRun 某个 step 的一次实际执行 状态 / agent / executor / summary
→
Transcript 这次 step run 的完整记录容器 不是 session 概念
→
TranscriptEvent message / tool / status / stdout / stderr append-only 事件流

为什么 StepRun 要独立

  • 当前 step 嵌在 workflowRun.steps[] 里，太适合展示，不适合承载过程明细
  • 后续 rerun / retry / resume 时，需要“step 的一次执行实例”概念
  • agent / executor / summary / transcript 都应归属于一次运行，而不是 step 定义

为什么 Transcript 比 Session 更纯

  • session 偏交互/传输/连接语义
  • transcript 偏业务记录语义，更贴近“查看 step 对话过程”
  • 后面即使增加输入能力，也只是 transcript 的写入方式变多，不会污染核心模型

2. 存储边界

编排摘要、运行实例、对话明细三层分离，避免一个对象既管状态又管历史大文本。

workflow_runs

保留工作流级状态与 step 摘要视图。

  • status
  • current_step
  • steps[] 轻量摘要

step_runs

承载某个 step 的一次实际执行。

  • workflow_run_id
  • step_id / step_name
  • agent_id / executor_type
  • transcript_id

transcripts + transcript_events

承载完整历史与增量读取。

  • append-only
  • 按 seq 增量拉取
  • UI 渲染成 chat
核心原则：workflow_runs 只管编排摘要，过程明细不再塞进 step.output 当唯一来源。

3. 不用 WebSocket 的实时方案

实时性来自增量轮询，不来自推送。传输层只是读取策略，不进入领域核心。

用户点击 Step 左侧选中一个 step
→
GET StepRun Detail 拿到 transcript_id / status / agent
→
GET Transcript Events 先拉完整历史或最近一段
如果 Step 仍在运行 每 1~2 秒轮询 after_seq
→
增量追加到右侧面板 形成“实时对话”体验
→
Step 完成后停止轮询 面板仍保留完整历史

4. 后端能力边界

抽的不是 session framework，而是“执行过程产出 transcript”的框架。

WorkflowService只做编排：启动 workflow、推进 step、更新 run 摘要
StepRunService创建/更新某个 step 的运行实例
TranscriptService创建 transcript、维护状态、查询 transcript
TranscriptEventServiceappend event、按 seq 增量查询
ExecutionEventSink给 executor 的统一写入口：appendMessage / appendToolCall / appendStdout / appendError
ExecutorAdapter把 Claude / Codex / OpenCode 的运行输出翻译成统一 TranscriptEvent
Polling API前端通过 after_seq 拉增量，不耦合传输协议
UI Projection前端按 event.kind 渲染 chat / tool card / status

6. 当前结论

这版已经按你的反馈修正了两个方向：去掉 WebSocket、把模型从 execution session 调整成更纯的 transcript 模型。

本轮推荐

  • 核心模型：WorkflowRun → StepRun → Transcript → TranscriptEvent
  • 读取方式：历史查询 + after_seq 轮询增量
  • 前端右侧：StepTranscriptPanel
  • 后端抽象：Transcript-producing execution framework

暂不进入核心模型的东西

  • WebSocket
  • Session 概念
  • Continue / resume protocol
  • 过度泛化 owner_type 平台
